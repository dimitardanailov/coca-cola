# Tickets Knowledge Structure

This folder is a ticket knowledge base.
Each ticket should keep execution and delivery context in one place: requirements, decisions, ticket-local diagrams, delivery notes, and optional code snippets.

## Naming Convention (Recommended)

Use a numeric sequence + ticket id for each ticket folder:

- `01.<ticket-id>`
- `02.<ticket-id>`
- `03.<ticket-id>`

Example:

- `01.2882103`
- `02.2884567`

Why this works:

- Numeric prefix keeps navigation stable and fast.
- Ticket id preserves traceability with Jira/Azure DevOps.
- Folder names stay short and grep-friendly.

## Ticket Folder Template

Inside each ticket folder, keep numbered markdown files so the story reads in order.

Suggested structure:

```text
tickets/
  01.2882103/
    ticket.meta.ts
    00.index.md
    01.requirements.md
    02.decisions.md
    diagrams/
    snippets/
    assets/
    demos/
```

Optional files (add only when the ticket needs them):

```text
    03.architecture.md
    04.implementation-plan.md
    05.testing.md
    06.release-notes.md
```

## Spec-Driven Ticket Flow (Default)

Use the execution-funnel approach from `MODEL_STRUCTURE.md` as the default flow for each ticket.

Recommended sequence:

1. Entry Gate and Scope Check in `00.index.md`
2. Spec in `01.requirements.md`
3. Evidence snapshot in `00.index.md` and/or `07.analysis.md`
4. Change Logic in `02.decisions.md`
5. Validation plan and evidence in `05.testing.md`
6. High-Risk Changes analysis in `03.architecture.md` or `07.analysis.md` when behavior-shaping systems are affected

Practical rule:

- Do not move implementation planning forward until scope, acceptance criteria, and evidence are explicit.

## File Purpose

### Required files

- `ticket.meta.ts`: typed metadata — status, owner, related docs and files.
- `00.index.md`: entry point, status, owner, knowledge nodes, relationship map, next steps, plus entry-gate/scope/evidence snapshot.
- `01.requirements.md`: scope, objective, inputs, acceptance criteria, constraints, and non-goals.
- `02.decisions.md`: decision log, alternatives considered, rejections, open questions, follow-up items.

### Folders

- `assets/`: ticket-local raw artifacts (uploads, screenshots, temporary exports). Use topic subfolders (e.g. `assets/arb/`).
- `diagrams/`: Mermaid, draw.io, or exported diagram images.
- `snippets/`: short code examples or commands relevant to the ticket.
- `demos/`: sprint demo artifacts for this ticket (slides, recordings, exported outputs). Add when a demo exists.

### Optional files (add only when the ticket needs them)

- `03.architecture.md`: design choices, trade-offs, sequence/data flow.
- `04.implementation-plan.md`: task breakdown and rollout steps.
- `05.testing.md`: test strategy, cases, validation evidence, known gaps.
- `06.release-notes.md`: what changed, impact, rollback notes.
- `07.analysis.md`: findings, dependencies, risks, assumptions, and high-risk before/after behavior analysis. Use when research is significant enough to deserve a separate file.

## Content Rules

- Keep facts, assumptions, and open questions clearly separated.
- For non-trivial tickets, structure conclusions as: facts, assumptions, unknowns, evidence, decision, next step.
- Avoid duplicating detailed info from root knowledge files; link to them instead.
- Put cross-file references in `00.index.md` for quick prompt context.
- Keep each markdown file focused on one concern.
- Update `lastUpdated` in `ticket.meta.ts` when status, owner, scope, or decisions change.

## External Artifacts (PDF, DOCX, Images)

- Store external files under `assets/` inside the ticket folder.
- Use topic subfolders when useful, for example `assets/arb/`.
- Keep canonical, reusable architecture artifacts in `docs/`; tickets should reference those canonical paths.
- For diagram-heavy architecture inputs, export to PNG and treat PNG as canonical for analysis and linking.
- PDF/DOCX may exist only as optional raw uploads; do not rely on them as the sole analysis source.
- Rename uploaded files to stable, readable names and keep numeric ordering:
  - `01-arb-architecture-overview.png`
  - `02-c4p-instance-strategy-page-01.png`
  - `03-network-boundary-diagram.png`
- Style guidelines for artifact names:
  - Use lowercase letters only.
  - Use kebab-case (`-`) between words.
  - Keep file names without spaces, brackets, or autogenerated IDs.
  - Prefix with two-digit order (`01-`, `02-`, `03-`) for predictable sorting.
- If the original filename carries useful metadata, preserve it in a note inside `00.index.md`.

## Prompt Usage

When asking AI assistants for help on a ticket, reference at minimum:

1. `tickets/<prefix.ticket-id>/00.index.md`
2. The specific working file (`01.requirements.md`, `03.architecture.md`, etc.)
3. Any root knowledge files that add context (`Profile.md`, `TeamMembers.md`, `TechnologyStack.md`)

## Optional Enhancements

- Add a lightweight checklist in `00.index.md` for progress tracking.
- Add a `Related Tickets` section in `00.index.md` when this ticket depends on or blocks another ticket.

## Ticket Metadata Policy

- Preferred format is `ticket.meta.ts` (use TypeScript, not YAML).
- Reason: readability plus light type safety with a consistent schema per ticket.
- No export/generation/conversion is required unless explicitly requested.
- No imports are required in metadata files.
- Duplication across ticket metadata files is acceptable when each ticket is a separate context.
- `owners` should use `Nickname` identifiers from `TeamMembers.md`.

### Metadata Field Rules

| Name | Type | Required | Allowed Values / Format | Purpose |
|---|---|---|---|---|
| TicketStatus | type alias | yes | `"todo" | "in-progress" | "blocked" | "done"` | Shared status vocabulary |
| TicketPriority | type alias | yes | `"low" | "medium" | "high"` | Shared priority vocabulary |
| TicketMeta.ticketId | string | yes | ticket id string | External tracking reference |
| TicketMeta.sequence | string | yes | two digits, for example `01` | Sorting and navigation |
| TicketMeta.title | string | yes | short descriptive title | Human-readable context |
| TicketMeta.status | TicketStatus | yes | see TicketStatus | Progress state |
| TicketMeta.priority | TicketPriority | yes | see TicketPriority | Delivery priority |
| TicketMeta.owners | string[] | yes | `Nickname` values from `TeamMembers.md` | Ownership |
| TicketMeta.relatedDocs | string[] | yes | relative paths to docs/assets | Source traceability |
| TicketMeta.relatedFiles | string[] | yes | relative paths in ticket folder | Knowledge graph links |
| TicketMeta.lastUpdated | string | yes | `YYYY-MM-DD` | Freshness tracking |

## Decision

For this repository, use:

- Folder format: `<two-digit-sequence>.<ticket-id>`
- File format inside ticket folder: `<two-digit-sequence>.<topic>.md`

This gives predictable sorting and consistent prompt context.
