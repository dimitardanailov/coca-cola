## Developer Explanation (2026-05-01, Why James Requested Format Change)

- Audience context:
  - This clarification is written for application developers and non-DevOps contributors.
  - Goal is to explain why the previous payload format was difficult for James to execute directly.
- What James clarified:
  - GitHub Actions Service Account must use predefined Google roles only.
  - Data Migration Service Account must use custom role permissions only.
  - GCP Migration Principal must use custom role permissions only.
  - Developer Principal must use custom role permissions only.
- Why the previous format was not ideal for execution:
  - Previous iterations mixed predefined roles and custom permissions for the same principal in multiple rows.
  - Mixed mode is ambiguous for implementation because it is unclear whether both should be applied or only one should be authoritative.
  - Ambiguity increases risk of permission drift and over/under-provisioning.
- Why this happened even while following the document:
  - The matrix structure includes both predefined-role rows and custom-permission rows.
  - Variable-level execution guidance was not fully documented in time, so mixed population looked valid from a contributor perspective.
  - James explicitly acknowledged missing detailed variable descriptions and then provided the principal-by-principal implementation model.
- What James found in this iteration:
  - Main issue was model alignment (how to apply), not necessarily permission intent (what capabilities are needed).
  - Requested action is to align each principal to one execution model and avoid mixed interpretation.
- Working rule for future updates:
  - Treat each principal as a single IAM contract type.
  - If principal is predefined-role based, populate predefined-role row only.
  - If principal is custom-permission based, populate custom-permission row only.
  - Do not mix both for the same principal unless James explicitly requests an exception.
