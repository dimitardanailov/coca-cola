# Decisions — 02.2954338 Google Cloud Permissions

## Decision Log

### [2026-04-24] Baseline Access Group Policy For Permissions Matrix

**Status**: accepted

**Context**
The permissions matrix file needed a clear, reusable access-control baseline before detailed API and IAM role mapping for NGPS/Promo+.

**Options Considered**

| Option | Pros | Cons |
|--------|------|------|
| Single table combining groups and environment permissions | Simple initial shape | Repeats group metadata many times; harder to scan |
| Split into group details + permission matrix tables | Better readability, less duplication, easier maintenance | Slightly longer section |

**Decision**
Use a split-table model at the top of the permissions file:
- Table 1: group details (`admin`, `developers`, `Operational`, members empty)
- Table 2: permission matrix by environment (Alpha, Betta, Gamma, Prod)

Use emoji status in the matrix (`✅` / `❌`) for fast visual scanning.

Baseline access rule:
- `admin`: read/write across all environments
- `developers`: read/write in Alpha/Betta, read-only in Gamma/Prod
- `Operational`: read-only across all environments

**Change Logic**
- Why this approach: Enforces least privilege, improves readability, and clearly protects higher environments.
- Alternatives rejected because: A single denormalized table duplicated metadata and made review slower.

**Consequences**
- NGPS/Promo+ role mapping now has a defined governance baseline.
- Gamma and Prod write access is explicitly restricted to `admin`, reducing operational and billing risk.
- Next iterations should focus on API enablement and predefined/custom role mapping per environment.

### [2026-04-24] Source Authorship And Style Preservation

**Status**: accepted

**Context**
The permissions matrix artifact in `promo-plus-core-infrastructure/.context/gcp-iam/apis-predefined-roles-custom-permissions.md` is authored by James Moorman. Promo Plus additions are needed for team readability and governance, but they must remain compatible with the source document style.

**Options Considered**

| Option | Pros | Cons |
|--------|------|------|
| Rewrite sections with a new style | Could optimize for one audience | Breaks consistency with source artifact and author expectations |
| Additive updates that preserve source style | Keeps continuity, easier review by source author and team | Requires stricter editorial discipline |

**Decision**
Use additive updates only, while preserving the structure and writing style established by James in the source artifact.

Scope boundary for this ticket:
- Keep James's existing document flow and naming.
- Add Promo Plus-specific readability sections only when they remain compatible with that style.

**Change Logic**
- Why this approach: Reduces review friction and prevents style drift in a shared cross-team artifact.
- Alternatives rejected because: A style rewrite would create unnecessary divergence from the source owner's document conventions.

**Consequences**
- Ticket outputs must explicitly note source authorship and style alignment.
- Future AI edits should be reviewed against style compatibility, not only technical correctness.
