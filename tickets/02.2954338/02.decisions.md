### [2026-04-24] Baseline Access Group Policy For Permissions Matrix

**Status**: accepted

**Context**
The permissions matrix file needed a clear, reusable access-control baseline before detailed API and IAM role mapping for NGPS/Promo+.

**Options Considered**

| Option | Pros | Cons |
|--------|------|------|
| Single table combining groups and environment permissions | Simple initial shape | Repeats group metadata many times; harder to scan |
| Split into group details + permission matrix tables | Better readability, less duplication, easier maintenance | Slightly longer section |

**Decision**
Use a split-table model at the top of the permissions file:
- Table 1: group details (`admin`, `developers`, `Operational`, members empty)
- Table 2: permission matrix by environment (Alpha, Betta, Gamma, Prod)

Use emoji status in the matrix (`✅` / `❌`) for fast visual scanning.

Baseline access rule:
- `admin`: read/write across all environments
- `developers`: read/write in Alpha/Betta, read-only in Gamma/Prod
- `Operational`: read-only across all environments

**Change Logic**
- Why this approach: Enforces least privilege, improves readability, and clearly protects higher environments.
- Alternatives rejected because: A single denormalized table duplicated metadata and made review slower.

**Consequences**
- NGPS/Promo+ role mapping now has a defined governance baseline.
- Gamma and Prod write access is explicitly restricted to `admin`, reducing operational and billing risk.
- Next iterations should focus on API enablement and predefined/custom role mapping per environment.

### [2026-04-24] Operator-First Scope Snapshot For AWS-to-GCP Migration

**Status**: accepted

**Context**
The current Infrastructure Services - Scope Snapshot is helpful but still module-centric. For migration execution, Admin needs an operator-first view that includes foundational domains beyond current root module wiring. The project also carries AWS legacy history via ngps-global and requires careful migration interpretation.

**Options Considered**

| Option | Pros | Cons |
|--------|------|------|
| Keep current snapshot focused only on wired OpenTofu modules | Simple and low-maintenance | Misses operational foundations and can block independent Admin operation |
| Expand snapshot to operator-first domains while keeping GCP target focus | Better run-time clarity and migration coverage | Requires additional curation and validation |

**Decision**
Expand Infrastructure Services - Scope Snapshot with operator foundations (including file storage, service principals/accounts, and logging/monitoring/reporting), while preserving GCP-target orientation.

Use ngps-global as migration-reference evidence only:
- It helps detect missing domains from AWS legacy behavior.
- It must not be treated as a direct permission template for GCP target operations.

**Change Logic**
- Why this approach: Improves operational independence for Admin and reduces reliance on ad hoc context escalation.
- Alternatives rejected because: A module-only snapshot can hide critical operational gaps in long-running migration programs.

**Consequences**
- Future snapshot updates must validate both migration completeness and operator usability.
- Admin workflows should avoid AWS-era access fallback assumptions.
