### [2026-04-27] `roles/resourcemanager.projectIamAdmin` Removed From Admin AD Group

**Status**: superseded

**Context**
`roles/resourcemanager.projectIamAdmin` was initially included in the admin AD group role list. This role allows any holder to set or replace any IAM policy on the project, including granting themselves `roles/owner`. It is a privilege escalation vector with no ceiling and is incompatible with a human-user AD group under any least-privilege or TCCC/C4P governance model.

Three converging constraints require its removal from human AD groups:

| Constraint | Source |
|---|---|
| TCCC/C4P policy — no admin for developers or tech leads | Sunil email, 2026-04-23 |
| DevOps model — resource provisioning is an automation function, not a human task | Vijay meeting, 2026-04-27 |
| Least-privilege — human admins have no legitimate need to rewrite project IAM policies | `01.requirements.md` constraints |

**Decision**
Initial proposal was to remove `roles/resourcemanager.projectIamAdmin` from the admin AD group. This proposal is now superseded and on hold.

- This role belongs exclusively on the **GitHub Actions CI/CD service account** (WIF identity), where OpenTofu applies IAM bindings during provisioning runs.
- Human admin AD group members operate services, not IAM policy infrastructure.
- Pattern evidence: MIXCODES completed sections in the permissions matrix already place this role on service accounts, not AD groups.

**Evidence**
- MIXCODES BETA/GAMMA/PROD service account ROLE arrays in `apis-predefined-roles-custom-permissions.md` include `roles/resourcemanager.projectIamAdmin` on the pipeline identity.
- No legitimate admin operator task requires directly modifying project-level IAM policy bindings outside of an IaC pipeline run.

**Consequences**
- No removal is currently treated as approved.
- Current source matrix remains at 24 admin roles until written governance outcome is received for the Alpha/Beta exception request.
- Final role model remains pending James/Vijay written response.

### [2026-04-27] `roles/serviceusage.serviceUsageAdmin` And `roles/iam.roleAdmin` Removed From Admin AD Group

**Status**: superseded

**Context**
Following the same governance review that removed `roles/resourcemanager.projectIamAdmin`, two further roles were identified as incompatible with a human AD group under TCCC/C4P policy and the IaC-first operating model.

- `roles/serviceusage.serviceUsageAdmin` enables and disables GCP APIs at the project level. This is infrastructure state — it is managed by `google_project_service` resources in OpenTofu and executed by the CI/CD pipeline. A human enabling or disabling APIs manually bypasses IaC governance and violates Sunil's explicit policy that the environment is governed through the GCP hierarchy and IaC.

Held by a human, this role allows crafting a role that grants any permission and assigning it — a privilege escalation path with the same risk profile as `roles/resourcemanager.projectIamAdmin`.

**Decision**
Initial proposal was to remove both roles from the admin AD group. This proposal is now superseded and on hold.

- Both belong exclusively on the **GitHub Actions CI/CD service account** (WIF identity) where OpenTofu manages API enablement and custom role definitions.
- Human admin AD group members operate deployed services, not infrastructure policy or configuration.

### [2026-04-24] Baseline Access Group Policy For Permissions Matrix

**Status**: accepted

**Context**
The permissions matrix file needed a clear, reusable access-control baseline before detailed API and IAM role mapping for NGPS/Promo+.

**Options Considered**

| Option | Pros | Cons |
|--------|------|------|
| Single table combining groups and environment permissions | Simple initial shape | Repeats group metadata many times; harder to scan |
| Split into group details + permission matrix tables | Better readability, less duplication, easier maintenance | Slightly longer section |

**Decision**
Use a split-table model at the top of the permissions file:
- Table 1: group details (`admin`, `developers`, `Operational`, members empty)
- Table 2: permission matrix by environment (Alpha, Betta, Gamma, Prod)

Use emoji status in the matrix (`✅` / `❌`) for fast visual scanning.

Baseline access rule:
- `admin`: read/write across all environments
- `developers`: read/write in Alpha/Betta, read-only in Gamma/Prod
- `Operational`: read-only across all environments

**Change Logic**
- Why this approach: Enforces least privilege, improves readability, and clearly protects higher environments.
- Alternatives rejected because: A single denormalized table duplicated metadata and made review slower.

**Consequences**
- NGPS/Promo+ role mapping now has a defined governance baseline.
- Gamma and Prod write access is explicitly restricted to `admin`, reducing operational and billing risk.
- Next iterations should focus on API enablement and predefined/custom role mapping per environment.
